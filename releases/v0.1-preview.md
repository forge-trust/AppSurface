# AppSurface v0.1.0 Release Preview

This is a preview of the first coordinated AppSurface release. AppSurface has not cut `v0.1.0` yet, so the final date, shipped scope, GitHub release URL, finalized changelog entry, and migration steps remain pending until the tag exists.

Use this page when you are deciding whether to install AppSurface packages from a prerelease feed or build from source before the first tag. Use [Unreleased](./unreleased.md) when you need the current merged-work ledger.

## Why this release matters

`v0.1.0` is the line where AppSurface becomes a coordinated package family instead of a set of useful repo-local building blocks. The release story is about trust: a first-time package consumer should be able to choose a package, understand the release risk, and know where migration guidance will live when the final tag ships.

The preview exists before the tag so package consumers can review the intended release shape without AppSurface pretending that `v0.1.0` has already shipped.

## Highlights

- A generated [package chooser](../packages/README.md) now points first-time adopters at the right public package, optional module, start-here README, and release surface.
- AppSurface owns a release hub, an unreleased proof artifact, a changelog, a pre-1.0 upgrade policy, and a tagged-release template.
- AppSurface Docs can render release trust metadata, provenance, Release page badges, search results, clean routes, and static exports for the release surface.
- Public package and CLI work is converging around a coordinated prerelease package set, including package metadata gates, artifact validation, and smoke-restore checks.
- The public `appsurface` tool now has a docs preview/export path for AppSurface Docs, while RazorWire keeps its package and CLI story explicit.

## Could this break me?

Yes, if you depend on prerelease packages, repo-local source, generated docs routes, CLI output, or public API names that are still moving before `v1.0.0`.

Known risk areas before the tag:

- Public Web and RazorWire enums now have explicit numeric values to protect consumers that persist or bind them.
- Browser status page APIs moved to the `BrowserStatusPage*` naming family before the first release.
- AppSurface Docs search-index payload APIs are internal implementation details; consumers should use the public docs search endpoint or their own search payload.
- AppSurface Docs route canonicalization now prefers clean public docs paths and redirects source-shaped Markdown paths.
- CLI and package publishing behavior is still prerelease-shaped until the coordinated release automation lands.

For standing compatibility rules, read the [pre-1.0 upgrade policy](./upgrade-policy.md).

## Package and API surface changes

The intended `v0.1.0` release is repository-wide. Public direct-install packages currently share one coordinated release story:

- `ForgeTrust.AppSurface.Core` for module composition, startup context, dependency graph orchestration, and shared process helpers.
- `ForgeTrust.AppSurface.Config` for typed configuration wrappers, source-aware audit reports, object/scalar validation, and environment override behavior.
- `ForgeTrust.AppSurface.Caching` for lightweight in-process memoization and cache policy primitives.
- `ForgeTrust.AppSurface.Auth` as a boundary-preview composition package for future AppSurface auth contracts; this preview registers options only and does not sign users in or enforce authorization.
- `ForgeTrust.AppSurface.Console` for command-first AppSurface console apps and CliFx-backed command hosting.
- `ForgeTrust.AppSurface.Dependency.Autofac` for Autofac-specific module registration.
- `ForgeTrust.AppSurface.Aspire` for Aspire app-model projects that want AppSurface-style modular composition.
- `ForgeTrust.AppSurface.Web`, `ForgeTrust.AppSurface.Web.OpenApi`, `ForgeTrust.AppSurface.Web.Scalar`, and `ForgeTrust.AppSurface.Web.Tailwind` for ASP.NET Core hosting, optional API/documentation UI, and Tailwind integration.
- `ForgeTrust.RazorWire` for reactive Razor fragments, server-sent updates, form failure UX, and package-owned UI contracts.
- `ForgeTrust.AppSurface.Cli` for repository-level AppSurface workflows such as docs preview and export.

Support runtime packages and proof-host packages remain part of the repository release stream, but they are not direct-install starting points for package consumers.

## CLI and docs-facing changes

The release surface is also an AppSurface Docs dogfood path:

- Release pages carry trust metadata that renders through AppSurface Docs.
- The package chooser is generated from `packages/package-index.yml`.
- AppSurface Docs can publish clean canonical docs routes while recovering source-shaped Markdown paths copied from GitHub or editors.
- The `appsurface docs` command can preview and export repository-owned AppSurface Docs surfaces.
- Static export and CI checks validate docs assets, managed links, contributor provenance, source links, and search behavior.

## When the tag exists

At tag cut:

1. Rename or copy `releases/v0.1-preview.md` to `releases/v0.1.0.md` and do the same for the sidecar.
2. Replace `Release Preview` title and trust status with final tagged-release language.
3. Fill the final version, date, shipped scope, GitHub release URL, changelog entry, and migration steps.
4. Remove `Facts pending until tag cut` or replace it with final provenance.
5. Update `CHANGELOG.md` to link to `releases/v0.1.0.md`.
6. Update `packages/package-index.yml` `release_notes_path` values from `releases/v0.1-preview.md` to `releases/v0.1.0.md`.
7. Regenerate and verify `packages/README.md`.
8. Update source links to the tagged note unless the docs route system needs a preview recovery alias.
